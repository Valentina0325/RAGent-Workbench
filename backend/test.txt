RAG（检索增强生成）是一种将信息检索与大语言模型生成相结合的技术。它通过从外部知识库中检索相关文档片段，然后将这些片段作为上下文提供给 LLM，从而生成更准确、更可靠的回答，有效减少模型幻觉。

ChromaDB 是一个开源的嵌入式向量数据库，专为 AI 应用设计。它支持多种向量索引算法（如 HNSW），提供简单的 API 用于存储、检索和过滤向量数据，非常适合本地开发和轻量级部署。

BM25 是一种用于信息检索的词袋排序函数，常用于搜索引擎中评估文档与查询的相关性。它基于 TF-IDF 并引入了文档长度归一化，能够更好地处理自然语言查询中的关键词匹配问题。

在 RAG 系统中，通常需要先将文档切分成小片段（chunk），然后用 embedding 模型将每个片段转换为向量，存入向量数据库。用户查询时，先将查询转换为向量，通过向量相似度检索找到最相关的片段，再将这些片段与原始查询一起提交给 LLM 生成最终答案。

混合检索（Hybrid Search）是指同时使用向量检索（语义相似度）和关键词检索（如 BM25）来召回文档，然后将两种结果按照一定权重合并排序。这种方法能兼顾语义理解与精确关键词匹配，显著提升检索准确率。

智谱 AI 的 GLM-4-Flash 模型是一款高效的中文大语言模型，支持 128K 上下文窗口，具备函数调用（Function Calling）能力，适合构建智能体（Agent）应用。同时，智谱还提供了 Embedding-2 模型，可将文本转为 768 维稠密向量。

一个完整的 AI 应用前端通常需要具备：流式对话（SSE）、多会话管理、文档上传与解析、图表可视化（ECharts）、性能优化（虚拟滚动）等模块。Vue 和 React 都可以胜任这些任务，其中 React 生态中的 Next.js 和 Vercel AI SDK 能大幅简化流式 AI 应用的开发。

当你开发 RAG 系统时，需要注意以下几点：切片尺寸（通常 256-512 token）、重叠窗口（避免语义断裂）、元数据保留（来源、页码）、多轮对话中的上下文管理、以及检索结果的排序与融合策略。

面试中如果被问到 RAG 的实现细节，可以围绕“文档处理 → 向量化 → 存储 → 检索 → 重排 → 生成”这个链路展开，并提及你使用的具体工具（如 ChromaDB、智谱 Embedding、混合检索）以及遇到的挑战（如大文件解析、检索延迟优化）。
